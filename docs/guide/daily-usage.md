# 日常使用

装好 ys-team 之后，你正常和 AI 对话就行。ys-team 在后台自动判断每个请求该走什么路径。

## 场景一：小修小补

> 你："帮我把 README 里的拼写错误修一下"

AI 判断这是个单文件、无风险的改动，直接修改。和没装 ys-team 一样，没有任何额外流程。

你会看到：
```
ys-team · trivial → 直接执行
```

什么时候会走这条路：单文件改动、没有跨模块影响、你明确说了是小改动。

## 场景二：明确的小范围改动

> 你："帮我把这三个 service 文件的日志格式统一成结构化日志"

AI 判断这是个范围明确、影响可控的改动（3 个文件、同一个模块内、不涉及 API 变更）。直接执行，但会在状态记录里留一笔。

你会看到：
```
ys-team · patch → 执行 + 留痕
```

改完后，`.ys_team/status.md` 里会多一行记录，方便以后追溯"这几个文件的日志格式是什么时候改的、为什么改的"。

什么时候会走这条路：改动不超过 3 个文件、影响在单模块内、意图清晰不需要讨论。

## 场景三：需要想清楚的改动

> 你："帮我给用户系统加一个注销功能"

AI 判断这是个跨模块的改动（可能涉及路由、session 管理、前端页面），需要先讨论清楚。

你会看到讨论过程：
```
**[主持人]** ys-team · spec-talk

角色简报：
- 架构角色：注销需要清理 session 和 token，建议同时处理"记住我"的 cookie
- 产品角色：注销后应该跳转到登录页，不是首页
- 交付角色：需要验证注销后无法用旧 token 访问受保护接口

决策：PASS
下一步：按规格执行
```

讨论收敛后，AI 会按照讨论产出的规格执行改动。执行完成后，会有独立的验收环节检查结果。

什么时候会走这条路：跨模块改动、有风险、需要讨论才能确定方案。

## 三种模式

ys-team 有三种工作模式，在项目的 `TEAM.md` 里配置：

- **manual** — 每个阶段都等你确认才继续。适合刚开始用，想看清楚每一步在做什么。
- **semi-auto** — 自动流转，但在"开始执行"和"提交代码"前会暂停等你确认。适合日常使用。
- **full-auto** — 全自动，只在出问题时才暂停。适合你对项目和 ys-team 都很熟悉之后。

刚开始建议用 manual 或 semi-auto，熟悉之后再切 full-auto。

## 和 AI 协作的技巧

- 如果你觉得 AI 判断的复杂度不对，直接说："这个直接改就行"或"这个需要先讨论一下"
- 讨论过程中如果你觉得方向不对，随时打断："不要这样做，换个方案"
- 如果执行结果不满意，验收环节会给出 REJECT，AI 会根据反馈重新执行
- 想看当前项目状态，对 Claude 说"看一下 ys-team 状态"

## 命令速查

| 你说什么 | 效果 |
|----------|------|
| "用 ys-team-init 初始化" | 首次启用工作流 |
| "用 ys-team-rebuild 重估" | 项目变化后更新配置 |
| "看一下 ys-team 状态" | 查看当前工作流状态 |
| "这是个小改动，直接改" | 跳过流程直接执行 |
| "这个需要先讨论一下" | 强制进入讨论流程 |
