# 案例与演进

## 真实案例

ys-team 用自己的方法管理自己。下面两个案例来自 ys-team 仓库的真实工作记录。

### 案例一：模板补全——为什么不能直接改

**问题**：项目切到全自动模式后，发现工作流需要的模板文件不完整。缺 4 个模板，涉及两个目录共 8 个文件。

**为什么不能直接改**：虽然"补模板"听起来简单，但涉及 8 个文件、两个目录，而且模板格式会影响后续所有项目的工作流。如果格式不对或者范围没控制好，后果会扩散。

**ys-team 在哪里帮了忙**：
- 讨论中发现了一个额外问题（另一个项目的格式和规范不一致），但明确标记为"超出本次范围，记录为后续追踪"，没有让范围膨胀
- 规格里写清了 Non-goals（不改已有模板、不改 init/rebuild skill），执行时不会越界
- 验收用了具体命令（`ls | wc -l` 和 `grep`），不是"看起来对了"

**结果**：3 个角色 1 轮讨论，范围清晰，执行无偏差，验收通过。

### 案例二：npm 发布遇到阻塞——"技术完成 ≠ 可交付"

**问题**：双模式安装（全局 + 项目级）的代码已经写完，准备发布到 npm。

**发生了什么**：交付守门人在验收时判定 BLOCKED——代码确实写完了，但 npm publish 需要账号凭证和 OTP 验证，这不是代码能解决的问题。

**为什么这很重要**：如果没有独立的验收环节，"代码写完了"很容易被当成"可以交付了"。交付守门人把"技术完成"和"可交付"区分开了，明确标注了阻塞原因和解除条件。

**结果**：BLOCKED 状态记录在案，等凭证问题解决后再继续。没有假装完成，也没有卡在那里不动。

## 定制

ys-team 的行为通过项目根目录的 `TEAM.md` 配置：

```yaml
mode: semi-auto          # manual / semi-auto / full-auto
spec_review_max_retries: 3
qa_max_retries: 3

roles:
  - 方法论架构师:     tools: [Read, Glob, Grep, Write, Edit]
  - 产品演进负责人:   tools: [Read, Glob, Grep, Write, Edit]
  - 交付守门人:       tools: [Read, Glob, Grep]
```

你可以调整：
- 工作模式（建议从 semi-auto 开始）
- 角色组合（根据项目需要增减）
- 角色的工具权限
- 重试上限

项目类型的差异（Python/Java/前端）由 init 自动处理，通常不需要手动调整。

## 演进方向

ys-team 还在持续改进：

- **跨角色记忆** — 让不同角色的经验可以流动，形成团队级知识
- **外部角色自动判断** — 根据改动类型自动决定是否需要安全审查、数据迁移审计等专业角色
- **更多平台支持** — 当前深度适配 Claude Code，未来计划支持更多 AI coding 工具

方法论的形式化规范见 [docs/methodology/](../methodology/)，欢迎参与讨论和贡献。
