# 为什么需要 ys-team

## AI 写代码的现状

AI coding agent 能力很强，但它有一个根本问题：它不知道什么时候该停下来想一想。

你说"加个接口"，它直接开始写代码。你说"重构一下"，它可能改了十几个文件。改完之后，你很难回答这些问题：

- 它为什么选了这个方案而不是另一个？
- 它改的范围合理吗？有没有改了不该改的？
- 如果出了问题，怎么回滚到改之前的状态？
- 下次遇到类似的需求，它会不会犯同样的错？

这不是 AI 的能力问题，是工作方式的问题。

## ys-team 的解法

给 AI agent 加一套工作纪律，核心是 3 个概念：

1. **规则**（rules.md）— 8 条禁止项，定义 AI 的行为硬边界
2. **现实索引**（reality.md）— 项目当前状态，AI 动手前先看这个
3. **Spec**（spec.md）— 复杂改动的执行合约，写清楚改什么、怎么验证

简单改动不需要流程。ys-team 自动判断复杂度，只在需要时启动完整流程。用户不需要知道什么时候该用规格、测试、审阅或诊断手法；这些判断属于 ys-team 的内部路由。

`.ys_team/` 把这套纪律落到项目本地：规则、现实索引、角色绑定、状态、模板和记忆都在仓库里。项目变化后可以 rebuild，让工作流重新贴合当前代码和文档现实。

## 反模式案例

### 案例 1：越界修改（行为边界）

**用户请求**："帮我把 UserService 的日志格式统一一下"

**没有约束时 AI 会怎么做**：改完 UserService 后，顺手把 OrderService、PaymentService 的日志也统一了。还重构了日志工具类，加了一个新的 LogFormatter。改了 12 个文件，你事后才发现。

**ys-team 的禁止项如何阻断**：
- 禁止项 #6（Style Ban：不顺手动别人的代码）— 自检锚点："每行 diff 能否追溯到用户请求？"OrderService 和 PaymentService 的改动追溯不到用户请求，被阻断。
- 禁止项 #4（Hard Ban：不越 Write-Scope）— 如果是 L2 spec，Write-Scope 只声明了 UserService 相关文件，其他文件的修改直接违反硬 gate。

**正确行为**：只改 UserService 的日志格式。如果发现其他服务也需要统一，在回复中提出建议，让用户决定是否扩大范围。

### 案例 2：无 spec 直接执行（流程边界）

**用户请求**："给用户系统加一个注销功能"

**没有约束时 AI 会怎么做**：直接开始写代码。改了 auth controller、session store、user model、前端路由、导航组件、API 文档，共 8 个文件。没有讨论 session 清理策略、是否需要通知其他服务、注销后的重定向逻辑。

**ys-team 的禁止项如何阻断**：
- 禁止项 #1（Hard Ban：未路由不改文件）— 路由判断发现这是跨模块改动（auth + user + frontend），判定为 L2。
- 禁止项 #3（Hard Ban：无 spec 不执行）— L2 改动未形成 spec.md，不得进入实现。自检锚点："是否存在对应 spec 目录且 spec.md 已写入？"答案是否，立即停止。

**正确行为**：路由判断为 L2 → 多角色讨论（session 清理策略、影响模块、回滚方案）→ 产出 spec.md（含 Write-Scope 和 Verification）→ 按 spec 执行。

### 案例 3：范围膨胀（Confirmation Ban）

**用户请求**："优化首页加载速度"

**没有约束时 AI 会怎么做**：从图片懒加载开始，然后发现 API 响应慢，于是加了缓存层。缓存需要 Redis，于是加了 Redis 依赖和配置。然后发现数据库查询也慢，于是加了索引。最后改了 15 个文件，引入了新的基础设施依赖。

**ys-team 的禁止项如何阻断**：
- 禁止项 #5（Confirmation Ban：范围扩大必须回头）— 执行到"加缓存层"时，发现需要修改 scope 外的文件（Redis 配置、数据库索引），必须停止执行，回到讨论。自检锚点："是否在继续执行前显式声明了 scope 变更并获得确认？"
- 这是 Confirmation Ban 而非 Hard Ban：显式声明变更并获得用户确认后可以继续，但不能静默扩大范围。

**正确行为**：按原始 spec 执行图片懒加载和前端优化。发现需要后端优化时，停止执行，向用户说明："发现需要扩大范围到缓存层和数据库索引，这超出了当前 Write-Scope。建议回到讨论，评估是否值得在本次一起做。"

### 案例 4：把流程负担丢给用户（产品边界）

**用户请求**："帮我把登录流程做得更可靠"

**没有约束时 AI 会怎么做**：反问用户要不要先写规格、要不要 TDD、要不要做审阅，或者直接让用户选择某个工作流。用户本来只是在描述目标，却被迫理解 agent 的内部工具菜单。

**ys-team 的正确行为**：先基于现实索引判断影响范围。如果只是文案或单点修复，直接改；如果涉及认证状态、会话、安全边界和回归风险，自动进入 L2，内部完成讨论、规格、执行和验证。用户只需要补充业务约束，不需要背流程。

## 四个原则

- 现实先于生成 — 不要凭假设写代码，先看项目实际长什么样
- 规格先于执行 — 复杂的事先写清楚要做什么，再动手
- 讨论归于收敛 — 讨论的目标是形成可执行的结论，不是无限展开
- 证据胜于感觉 — 交付结果必须可验证、可追溯

## 适合谁

- 个人开发者用 AI coding agent 写代码，想让 AI 更靠谱
- 小团队共用 AI agent，需要基本的协作纪律
- 任何希望给 AI 编码建立最低治理约束的场景

对已有成熟工程治理的大团队，ys-team 更适合作为 AI 协作层的补充。
