# 为什么需要 ys-team

## AI 写代码的现状

AI coding agent 能力很强，但它有一个根本问题：它不知道什么时候该停下来想一想。

你说"加个接口"，它直接开始写代码。你说"重构一下"，它可能改了十几个文件。改完之后，你很难回答这些问题：

- 它为什么选了这个方案而不是另一个？
- 它改的范围合理吗？有没有改了不该改的？
- 如果出了问题，怎么回滚到改之前的状态？
- 下次遇到类似的需求，它会不会犯同样的错？

这不是 AI 的能力问题，是工作方式的问题。一个人类工程师也不会拿到需求就直接写代码——他会先看看现有代码长什么样，想清楚该怎么改，改完了跑一下测试。

## ys-team 的解法

ys-team 给 AI agent 加了一套工作纪律，核心路径是四步：

1. 先对齐现实 — AI 先读你的项目结构和模块关系，知道自己在改什么
2. 讨论收敛 — 复杂的改动，多个角色从不同角度审视（架构合理吗？交付完整吗？有风险吗？），形成明确结论
3. 按规格执行 — 讨论产出的规格就是执行合约，AI 只能改规格里声明的文件
4. 证据验收 — 改完了用可执行的命令验证，不是"我觉得改好了"

简单的改动不需要走这个流程——ys-team 会自动判断复杂度，只在需要的时候才启动完整流程。

## 四个原则

- 现实先于生成 — 不要凭假设写代码，先看项目实际长什么样
- 规格先于执行 — 复杂的事先写清楚要做什么，再动手
- 讨论归于收敛 — 讨论的目标是形成可执行的结论，不是无限展开
- 证据胜于感觉 — 交付结果必须可验证、可追溯

## 适合谁

ys-team 更适合：

- 个人开发者用 Claude Code 写代码，想让 AI 更靠谱、更可控
- 小团队共用 AI agent，需要基本的协作纪律
- 任何希望给 AI 编码建立最低治理约束的场景

对已有成熟工程治理的大团队，ys-team 更适合作为 AI 协作层的补充，而不是替代现有流程。

## 和直接用 Claude Code 有什么不同

没有 ys-team 时：
> 你："给用户系统加一个注销功能"
> AI：直接开始写代码，改了 8 个文件，你事后才知道它改了什么

有 ys-team 时：
> 你："给用户系统加一个注销功能"
> AI：先看项目结构，判断这是个需要讨论的改动 → 多角色讨论（这个功能影响哪些模块？session 怎么处理？需要通知其他服务吗？）→ 形成明确规格（改哪些文件、怎么验证、怎么回滚）→ 按规格执行 → 用命令验证结果

区别不是"多了流程"，而是"AI 在动手之前先想清楚了"。
