# 角色协议

## 角色卡结构

每个角色定义为一张角色卡，包含以下字段：

| 字段 | 必需 | 说明 |
|------|------|------|
| id | 是 | 角色标识符（英文，如 spec-reviewer） |
| title | 是 | 角色显示名（可本地化） |
| subagent_policy | 是 | `required`、`preferred` 或 `local_only` |
| 职责摘要 | 是 | 1-2 句话说明角色做什么 |
| 详细职责 | 否 | 具体职责列表 |
| 输入 | 否 | 角色工作时读取的制品 |
| 输出 | 否 | 角色工作后产出的制品 |
| 约束 | 否 | 角色的权限边界和行为限制 |

## 角色分类

### 内部角色（local_only）

始终可用，不需要额外审批。项目初始化时根据项目类型生成。

基线提供 6 个内部角色模板：

| 角色 | 职责 | 参与阶段 |
|------|------|----------|
| 方法论架构师 / project-architect | 方法论定义、原则收口、边界控制 | spec-talk |
| 产品演进负责人 / domain-integrator | 功能规划、用户体验、接入路径 | spec-talk |
| 文档规格管理员 / doc-spec-steward | 文档权威、spec 模板、同步规则 | spec-talk |
| 交付守门人 / delivery-guard | spec 完整性、证据、回滚、交付闭环 | spec-talk, close |
| 规格审阅人 / spec-reviewer | 独立审阅 spec 质量 | spec-review |
| 质量保障守门人 / qa-guard | 独立验证落地效果 | qa |

项目可根据实际需要增减内部角色。

### 外部角色（preferred）

需要用户审批才能加入讨论。基线提供 4 个外部角色候选：

- 安全审查顾问 / security-reviewer
- 数据迁移审计顾问 / db-migration-auditor
- 可观测性与性能顾问 / observability-reviewer
- 前端可用性顾问 / frontend-accessibility-reviewer

## 角色选择规则

每次讨论开始时，根据改动类型选择参与角色：

- 选择规则由项目团队配置定义
- 至少选择一个角色
- 选择依据是改动涉及的领域，而非固定组合

示例规则：
- 涉及架构或原则变更 → 必须包含架构角色
- 涉及功能规划或用户体验 → 必须包含产品角色
- 涉及交付验收 → 必须包含交付角色
- spec 完成后独立审阅 → 使用审阅角色
- 落地验收 → 使用质检角色

## 上下文隔离原则

角色之间通过制品文件通信，不共享对话历史。这是状态机设计的核心约束。

具体表现：
- spec-review 角色只看到 requirement.md 和 control.md，看不到 spec-talk 的讨论过程
- qa 角色只看到 spec 制品和实现结果，不参与执行决策
- 每个角色启动时读取自己的角色记忆，不读取其他角色的记忆

这确保审阅和质检的独立性，避免群体思维。

## 工具绑定

角色可以绑定项目已内化的工具能力：

- 工具绑定在团队配置中声明
- 绑定的工具在角色对应的工作流环节自动建议使用
- 工具绑定由初始化/重估流程写入

## Subagent Policy 语义

角色的 `subagent_policy` 字段声明该角色是否适合用独立 subagent 运行：

| 值 | 含义 | 典型场景 |
|----|------|----------|
| `required` | 必须用独立 subagent | 需要上下文隔离的审阅、质检角色 |
| `preferred` | 边界清晰或需要并行时优先用 subagent | 外部角色、独立执行的角色 |
| `local_only` | 保持在主会话中 | 需要与用户直接交互的角色 |

这是提示而非强制——上层编排约束（如编排模式配置）优先级更高。

## 外部角色升级协议

当讨论中发现需要外部角色的专业能力时：

1. 停止继续扩展讨论
2. 输出外部支持请求，必须包含四个要素：
   - 为什么内部团队能力不足
   - 需要哪个外部角色
   - 外部角色需要回答什么问题
   - 不引入外部角色的风险
3. 等待用户同意后再加载外部角色继续

未来方向：基于改动类型和历史数据自动判断是否需要外部角色，减少人工审批。

## 角色演进

### 角色缺口检测

在讨论过程中，如果发现以下情况，记录演进请求：

- 讨论涉及某个领域（安全、数据迁移、可访问性）但没有对应角色
- 结果卡出现"外部支持请求"
- 讨论因缺少专业能力而阻塞

### 工具缺口检测

在执行过程中，如果发现以下情况，记录演进请求：

- 执行需要某种方法（TDD、设计规范、性能基准）但工具箱中没有
- 角色的工具绑定为空但执行明显需要工具支持

### 演进请求格式

```
标识: EVO-YYYY-MM-DD-N
日期: YYYY-MM-DD
类型: role / tool
来源: spec-talk / spec-work
关联 spec: <当前 spec 标识>
缺口描述: <缺少什么能力>
发现者: <哪个角色>
建议: <推荐的外部角色或工具>
状态: pending
```

演进请求在下次重估时评估，决定是否内化。

## 角色记忆

每个角色维护独立的跨任务记忆：

- 角色启动时读取自己的记忆文件
- 工作结束时回顾本次经验，有新学习则写入
- 记忆有容量上限，超出时压缩
- 角色记忆不跨角色共享（当前设计）
