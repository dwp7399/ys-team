# 初始化与演进

## 初始化哲学

- 最小有用基线：生成项目能用的最小 `.ys_team/` 结构，不强制完整
- 先读现实再生成：扫描项目实际文件和结构，不凭假设生成
- 不要求用户理解内部结构：用户只需知道"初始化一次"，不需要理解角色、模板、策略的内部关系

## 初始化产出

初始化必须产出以下类别的内容：

| 类别 | 说明 |
|------|------|
| 团队配置 | 角色组合、工具绑定、选择规则 |
| 角色定义 | 内部角色卡（根据项目类型选择） |
| 方法配置 | 弱引导原则、讨论透镜 |
| 治理策略 | 执行规则、文档权威、同步规则 |
| Spec 模板 | control / work / requirement / workspace / review / qa-report |
| 状态追踪 | 初始状态文件 |
| 记忆结构 | 角色记忆目录和策略文件 |
| 现实索引 | 模块级索引（关系图 + 业务摘要） |
| 项目配置 | 编排模式、重试上限、记忆配额 |

## 项目类型检测

初始化时扫描项目文件，分类项目类型：

| 检测信号 | 项目类型 |
|----------|----------|
| requirements.txt / pyproject.toml | python-backend |
| pom.xml / build.gradle | java-backend |
| package.json + React | frontend-react |
| 前后端都有 | fullstack |
| 以上都不匹配 | general |

项目类型影响：
- 推荐内化的工具
- 角色卡的具体职责描述
- 现实索引的结构和粒度

## 语言检测与本地化

初始化检测用户语言偏好，生成对应语言的：
- 角色名称和职责描述
- 团队配置
- 模板内容

## 现实索引生成

现实索引采用关系优先、摘要辅助、规模自适应的策略：

- 关系优先：先建立模块间的依赖关系图
- 摘要辅助：每个模块 2-3 句业务职责摘要
- 规模自适应：小项目用核心模块索引，大项目分层

## 工具箱内化流程

1. 根据项目类型匹配推荐工具
2. 检查推荐工具是否已安装
3. 选择性提取相关部分（不全量复制）
4. 适配到项目方法论表述
5. 绑定到对应角色
6. 记录来源、版本和重新内化触发条件

## 演进触发条件

以下情况触发重估（rebuild）：

- 项目技术栈发生变化
- 项目新增或移除了主要模块
- 现有角色不再匹配交付现实
- 模板不再匹配验收需要
- 文档权威声明与实际不符
- 基线版本有重大更新

## 重估规则

- 改最小面：只更新确实需要变化的部分
- 保留本地化：不覆盖项目已定制的内容
- 版本对齐：检查项目版本与基线版本的差异
- 工具重新评估：检查候选列表、现有工具的"需要深化"标记、项目类型变化后的新推荐
- 记忆健康检查：确保角色记忆文件与当前角色对齐

## 版本管理

三个版本号各自独立，追踪不同的变化面：

| 版本 | 文件 | 追踪什么 | 变更触发 |
|------|------|----------|----------|
| 方法论版本 | `docs/methodology/VERSION` | 方法论规范本身 | 状态机、角色协议、制品 schema 等规范变更 |
| 基线版本 | `examples/baseline/.ys_team/VERSION` | 默认工作流骨架 | 模板、角色卡、策略文件等基线内容变更 |
| 项目版本 | `.ys_team/VERSION` | 项目本地基线 | 项目执行 init 或 rebuild 时写入 |

关系：
- 基线版本和项目版本跟随 npm 包版本（当前 0.3.x）
- 方法论版本独立编号（当前 1.0.0），因为方法论规范是新增的独立产出
- rebuild 时对比项目版本与基线版本，决定是否需要更新
- 方法论版本变更不一定触发基线版本变更（规范澄清不影响模板）

## 发布线治理

当仓库将 baseline 作为 npm 包主体一并发布时，建议显式区分两条线：

- 发布线：`package.json`、bundled baseline 的 `.ys_team/VERSION`、仓库自用 `.ys_team/VERSION`
- 方法论线：`docs/methodology/VERSION`

发布线约束：

1. 发版工作在 `release/<version>` 分支完成
2. 真实发布成功（如 `npm publish`）后，发布分支才合并回 `main`
3. 合并回 `main` 后创建 git tag `<version>`
4. 发版类 spec 只有在发布成功且代码回到 `main` 后才 close

如果仓库自用 `.ys_team/VERSION` 低于当前发布线，说明仓库自身尚未完成一次与当前 baseline 的同步。

## Baseline changelog 维护

当 baseline 被持续演进并对外分发时，应维护一份结构化 changelog，帮助用户理解“版本差异意味着什么”。

建议规则：

1. changelog 与 baseline 一起版本化，随重大变更同步更新
2. 每个版本段落至少包含：
   - `What Changed`
   - `Why`
   - `Key Files`
   - `Migration Hint`
3. 未发布版本可使用 `Unreleased` 段持续累积
4. `check-update` 一类升级提示能力，应优先从 changelog 提取摘要，而不是硬编码版本说明

这样做的目标不是记录所有提交，而是记录对项目使用者真正有迁移价值的基线变化。
