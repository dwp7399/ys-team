# ys-team 方法概述

## 定位

ys-team 是一种 AI 团队治理方法，以共享现实索引为对齐基础、以多角色讨论为收敛机制、以 Spec 为执行与验收统一依据。

它不是固定流程图，不是项目管理工具，不是重型治理平台。它提供的是一组稳定约束，让 AI agent 在帮你写代码时不乱来。

## 四个不变量

1. 现实先于生成 — 项目必须维护能反映真实状态的共享现实索引，所有决策基于当前现实而非假设。
2. 规格先于执行 — 非 trivial 工作先收敛为 Spec，再依据 Spec 执行和验收。
3. 讨论归于收敛 — 多角色讨论的目标是形成可执行的结论，不是无边界扩展。
4. 证据胜于感觉 — 交付结果必须可验证、可追溯，验收基于证据而非主观判断。

## 弱引导、强结果

ys-team 不规定每一步怎么做，而是约束结果必须满足什么条件。

含义：
- 先理解当前在讨论什么（方法论、skill 边界、仓库接入、还是 spec 执行），再选择路径
- 不把方法论讲成固定流程图
- 当问题已经收敛到可执行程度时，默认继续进入 spec / work / acceptance 链，而不是停在抽象讨论
- 每轮讨论都要收敛到明确结论、现状和下一步

## 目标受众

- 使用 AI coding agent 的开发者和团队
- 希望 AI 在结构化约束下工作，而不是自由发挥
- 项目规模从个人到小团队

## ys-team 不是什么

- 不是 CI/CD 工具 — 它不替代构建、测试、部署流水线
- 不是项目管理软件 — 它不做甘特图、看板、工时统计
- 不是代码审查工具 — 它的 spec-review 是对规格的审阅，不是对代码的 review
- 不是强制流程 — 它允许项目本地化，不强迫所有仓库长成同一种样子

## 用户心智

对外只保留很少的用户心智：

1. 安装 ys-team skills
2. 在项目里执行一次初始化
3. 项目形态明显变化后，再执行一次重估

不要把内部实现细节包装成用户必须理解的流程图。
