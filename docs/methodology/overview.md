# ys-team 方法概述

## 定位

ys-team 是一组让 AI agent 在帮你写代码时不乱来的纪律约束。

它不是项目管理工具，不是重型治理平台，不是固定流程图，也不是需要用户手动选择的一组工作流菜单。它提供的是项目本地治理层：用户照常对话，ys-team 在内部完成路由、讨论、规格、执行、验证和收口，让 AI 的行为可预测、可追溯、可验收。

## 四个不变量

1. **现实先于生成** — 项目必须维护能反映真实状态的共享现实索引，所有决策基于当前现实而非假设。
2. **规格先于执行** — 非 trivial 工作先收敛为 Spec，再依据 Spec 执行和验收。
3. **讨论归于收敛** — 多角色讨论的目标是形成可执行的结论，不是无边界扩展。
4. **证据胜于感觉** — 交付结果必须可验证、可追溯，验收基于证据而非主观判断。

## 用户不背流程

ys-team 可以吸收外部 agent workflow 的好模式，但吸收结果必须折叠进内部路由和 baseline，而不是变成用户要记住的新入口。

- 用户表达目标，ys-team 判断 L0/L1/L2。
- 内部可以使用澄清、规格、测试、审阅、诊断、架构复盘等策略。
- 对外只暴露稳定结果：直接改、留痕、形成 spec、执行验证、收口。
- 项目结构和协作约定变化时，通过 rebuild 重估 `.ys_team/`，而不是要求用户重新学习流程。

这条规则优先于引入新方法。凡是会增加用户主动选择负担的模式，都不能直接外显。

## 反向控制（8 条禁止项）

禁止项比推荐项约束力更强。以下 8 条是 ys-team 的行为硬边界。

| # | 类别 | 级别 | 口号 | 行为 | 自检锚点 |
|---|------|------|------|------|----------|
| 1 | 流程 | Hard Ban | 未路由不改文件 | 每个请求先完成 L0/L1/L2 判断，未判断前不得修改任何文件 | 回复中是否出现路由判断标志？ |
| 2 | 行为 | Hard Ban | 未读现实不下结论 | 涉及影响范围、风险或方案选择时，必须先读现实索引或相关代码 | 结论中引用的模块/文件，是否在本轮对话中实际读取过？ |
| 3 | 流程 | Hard Ban | 无 spec 不执行 | L2 改动未形成 spec.md，不得进入实现 | 是否存在对应 spec 目录且 spec.md 已写入？ |
| 4 | 流程 | Hard Ban | 不越 Write-Scope | spec-work 只能修改 Write-Scope 声明的文件 | 本次 diff 中每个文件路径是否在 Write-Scope 列表内？ |
| 5 | 行为 | Confirmation Ban | 范围扩大必须回头 | 发现必须修改 scope 外文件时，停止执行，回到讨论 | 是否在继续执行前显式声明了 scope 变更并获得确认？ |
| 6 | 行为 | Style Ban | 不顺手动别人的代码 | 不重构无关代码、不格式化无关文件、不补未要求的功能 | 每行 diff 能否追溯到用户请求或 spec 条目？ |
| 7 | 流程 | Hard Ban | 无证据不说完成 | 无 verification 结果、evidence 或明确"无法验证"说明，不得声明完成 | close/done 前是否有可复核的验证产出？ |
| 8 | 流程 | Hard Ban | 文档不同步不收口 | 受影响文档未同步更新，不得 close | Write-Scope 中的文档类文件是否全部已更新？ |

### Ban Levels

| 级别 | 含义 | 违反时行为 |
|------|------|------------|
| Hard Ban | 硬 gate | 立即停止，必须先满足条件 |
| Confirmation Ban | 软 gate | 显式声明并获得确认后可继续 |
| Style Ban | 默认禁止 | 用户明确要求时可以做 |

### Escape Clause

L0 级别请求可跳过 #1 和 #2，但必须声明跳过原因。

## 三级路由

所有改动请求先经过路由判断，按影响范围分为三个级别：

| 级别 | 名称 | 判定条件 | 流程 | 产出 |
|------|------|----------|------|------|
| L0 | trivial | 单文件、无影响、用户声明 trivial | 直接执行 | 无 |
| L1 | patch | ≤3 文件、单模块内、意图清晰 | 执行 → 留痕 | patch-note |
| L2 | spec | 跨模块、有风险、需讨论收敛 | 完整状态机 | spec + evidence |

**举证责任在"降级"：不确定时走 L2。**

L2 内部生命周期可以理解为：Define（澄清目标）→ Plan（形成 spec）→ Build（按 Write-Scope 执行）→ Verify（收集证据）→ Review（独立审阅/质检）→ Ship（状态收口）。这些是编排层语言，不是用户命令。

## Spec 最小定义

Spec 是执行合约。一个 spec.md 必须包含：

- **Background** — 为什么要做
- **Goals / Non-goals** — 做什么、不做什么
- **Deliverables** — 具体交付物
- **Write-Scope** — 允许修改的文件范围（执行时的硬边界）
- **Acceptance Criteria** — 验收标准
- **Verification** — 可直接执行的验证命令

可选：Risks、Rollback Plan、Depends-On。

Spec 目录结构：

```
docs/specs/<phase>/<id>/
├── spec.md          # 执行合约（必须）
├── checklist.md     # 交付检查清单（必须）
├── evidence/        # 验收证据（必须）
├── work.md          # 执行日志（spec-work 维护）
├── work-XX.md       # 多步拆分子合约（可选）
├── review.md        # 审阅结论（semi-auto/full-auto）
└── qa-report.md     # QA 报告（semi-auto/full-auto）
```

目录生命周期：

- `queued/`：spec 已起草，尚未通过独立审阅，不能进入实现。
- `active/`：spec 已进入执行主链，承载 `spec-review / spec-work / qa / close` 中的当前工作。
- `completed/`：close 完成后的短暂停留区，可直接继续迁入 `archive/`。
- `cancelled/`：明确放弃、不再继续推进的 spec。
- `archive/`：历史已完成 spec 的长期归档位置，保留原制品，不为归档补格式。

`work.md` 的语义固定为执行期日志：记录 `spec-work` 阶段的关键决策、偏差处理和验证进展。它不是讨论纪要，也不是每个阶段都要单独生成的新模板。

`status.md` 的语义固定为当前快照：记录活跃 spec、阻塞项、待办和最近 10 条判断。跨月的轻量统计进入 `.ys_team/history/YYYY-MM.md`，不再继续堆在 `status.md`。

## 角色与记忆

ys-team 固定的是治理槽位，不是固定角色名。默认槽位写在 `.ys_team/config.yaml` 的 `governance_slots` 中，项目当前启用的角色仍写在 `roles` 中，而槽位到角色的实际绑定写在 `slot_bindings` 中。

默认外部角色池来源写在 `.ys_team/role-pool.yaml`。它是一个钉住版本的离线参考清单：记录来源仓库、候选角色、治理槽位和按项目类型的默认映射，供 `init/rebuild` 生成本地绑定时使用。

角色的价值在于 **上下文隔离**：讨论时用独立 subagent 保证多视角，审阅和质检角色看不到讨论过程，只看制品。

### 角色记忆

每个角色维护独立的错题本式记忆（`.ys_team/memory/<role-id>.md`）：

- **摘要头**：当前核心原则（rebuild 时从条目自动提炼）
- **经验条目**：错误模式 → 正确做法 → 适用场景
- 启动时先读摘要头，需要细节时读具体条目
- 条目超过阈值时，rebuild 时合并相似条目

### 临时角色

config 里未列出但讨论中需要的角色，按需创建，并以 `slot_bindings.binding_type: temporary` 标记。如果被反复引入（rebuild 时检测），建议升级为稳定绑定。

## 交付检查清单

每个 spec 从模板复制一份 `checklist.md`，绑定 5 个核心流程：

1. **spec-talk** — 讨论收敛，spec.md 写入
2. **spec-review** — 独立审阅 spec 质量
3. **spec-work** — 按 spec 执行实现
4. **qa** — 独立验证落地效果
5. **close** — 提交、更新状态、归档

每个阶段完成时打勾，确保交付主链不断裂。`close` 阶段至少要回写状态、完成最小 Git 收口，并把 spec 迁到正确目录。

## 用户心智

对外只需要理解 3 个概念：

1. **规则**（rules.md）— AI 的行为边界
2. **现实索引**（reality.md）— 项目当前状态
3. **Spec**（spec.md）— 复杂改动的执行合约

操作只需要 2 步：

1. 安装 ys-team skills，在项目里执行一次初始化
2. 项目形态明显变化后，再执行一次重估

项目可以额外维护领域语言说明、ADR、issue 约定或团队协作约定。ys-team-init/rebuild 可以识别并参考这些材料，但默认不覆盖本地定制，也不把它们变成每个项目必须具备的重产物。

## ys-team 不是什么

- 不是 CI/CD 工具 — 不替代构建、测试、部署流水线
- 不是项目管理软件 — 不做甘特图、看板、工时统计
- 不是强制流程 — 允许项目本地化，不强迫所有仓库长成同一种样子
