# 治理规则

## 执行规则

- 改动按三级分流：L0 trivial（直接执行）、L1 patch（执行 + 留痕）、L2 spec（完整流程）
- L2 改动必须先有 spec，再按 spec 执行和验收
- 文档必须反映项目当前真实状态
- spec、实现和验收证据应保持同一次交付闭环

## 文档权威

项目应声明哪些文档承载什么权威：

- 哪些文档承载当前现实（现实索引）
- 哪些文档承载接口或结构说明
- 哪些文档承载 roadmap / spec
- 哪些文档是代码变化后必须同步更新的权威来源

文档权威声明避免"到处都有一点，哪里都不完整"的状态。

## 文档同步规则

实现变化时，相关文档必须同步更新：

- 模块边界变化 → 更新现实索引
- 对外行为变化 → 更新用户文档
- 方法论定义变化 → 更新方法论文档和相关 skill 定义
- spec 模板变化 → 更新模板文件

同步更新应在同一次交付中完成，不留"下次再补"的债务。

## 现实索引

现实索引是 AI 团队各角色对项目模块现实达成共识的基础设施。

- 承载内容：模块划分、业务职责摘要（2-3 句）、模块间依赖关系
- 维护时机：模块新增、模块职责重大变化、模块间依赖关系变化时更新
- 使用时机：讨论改动前先读索引判断影响范围，执行时对齐 Write-Scope 语义
- 读取优先：在讨论改动前，先读索引，再决定是否需要深入代码

## 交付门禁

以下情况需要更严格的交付复核：

- 数据迁移
- 对外接口行为变化
- 多模块联动
- 外部回调或异步链路上线
- 安全敏感操作

项目可根据实际情况扩展门禁条件。

## 排他工作流

ys-team 启用后，工作流具有排他性：

- 禁止自动触发工具箱以外的能力
- 已内化的工具清单由工具箱索引维护
- 仅内化后的版本在工作流中可用
- 用户显式调用未内化能力时，记录使用行为作为下次重估的内化候选
- 用户在项目配置中显式声明的全局能力不受此规则限制

## 工具箱治理

### 内化

将外部能力选择性引入项目工具箱：

- 只提取与项目相关的部分，不全量复制
- 适配到项目方法论的表述方式
- 绑定到对应角色
- 记录来源和版本

### 候选追踪

用户显式使用但未内化的能力，记录到候选列表，下次重估时评估。

### 重新内化触发

以下情况触发重新内化评估：

- 原始能力有重大更新
- 项目需求变化导致当前内化版本不够用
- 角色反馈工具不好用

## 记忆治理

### 角色记忆

- 每个角色只读写自己的记忆文件，互不可见
- 记录跨任务的经验积累，不记录当前任务临时状态
- 有容量上限，超出时压缩（保留相关经验、合并相似条目、删除过时条目）
- 写入判断标准：是否发现非显而易见的陷阱或模式？是否有未来同类任务需注意的点？

### 记忆隐私约束

- 不记录个人身份信息（PII）
- 不记录外部系统的密钥、token 或凭证
- 只记录技术经验和项目特有规律

### 工作记忆（workspace.md）

- 跟随 spec 生命周期，位于 spec 目录下
- 记录当前任务的实时状态、阻塞、决策、阶段间传递的上下文
- 有容量上限，超出时压缩
- spec 关闭后随 spec 目录归档

### 记忆健康检查

重估时检查：
- 角色记忆文件是否超限
- 是否有记忆文件对应的角色已移除（标记归档，不删除）
- 是否有新角色缺少记忆文件（自动创建）

## 并行策略

### Spec 级并行

多个不相关的 spec 可同时执行。

- 冲突检测：如果两个 spec 的 Write-Scope 涉及同一模块，降级为串行
- 每条并行线在状态追踪中独立记录

### 角色级并行

同一 spec 内多个角色可并行工作。

- 角色间有依赖时仍需串行
- 讨论收敛后汇总各角色结果

### 默认关闭

并行能力默认关闭。项目在确认以下条件后可开启：
- 团队对工作流已熟悉
- 状态追踪正常运行
- 项目模块边界清晰
