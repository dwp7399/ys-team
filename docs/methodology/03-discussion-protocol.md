# 讨论协议

## 意图优先

讨论开始前，先做三段判断：

1. 当前对象是什么 — idea / spec / requirement / roadmap / 已有制品
2. 当前目标是什么 — 澄清 / 起草 / 审阅 / 重新定义 / 执行准备 / 交付判断
3. 当前阻塞是什么 — 边界不清 / 缺少验证 / 依赖未完成 / 需要外部支持 / 无阻塞

三段判断完成后，再选择输出路径：

- initiative 级别（新功能、重构、架构变更）→ 产出 spec
- stage 级别（版本规划、阶段目标）→ 产出 roadmap
- 澄清级别（理念、边界、方法论问题）→ 不产出制品

如果三段判断中任何一项不清楚，先向用户确认，不要猜测。

## 讨论流程

1. 确定讨论意图和范围
2. 根据改动类型选择参与角色（参考角色选择规则）
3. 各角色基于现实索引和已有信息给出初始判断
4. 识别分歧点和风险
5. 收敛讨论，形成结论
6. 如需外部角色，暂停并请求用户审批
7. 输出结果卡和角色简报
8. 如产出 spec，写入需求记录和工作记忆

## 收敛规则

讨论的目标是收敛，不是无边界扩展。

- 每轮讨论必须收敛到明确结论、现状和下一步
- 当问题已经收敛到可执行程度时，默认进入 spec / work / acceptance 链
- 不在抽象讨论中停留过久

### 讨论成本意识

讨论成本是自检信号，不是硬限制。每轮讨论后，检查对话是否仍在收敛。

警告信号：
- 重复出现前轮已有的论点
- 在前一轮阻塞点未解决的情况下引入新外部角色
- 在扩大范围而非收窄到决策

出现以上信号时，优先停轮，输出当前状态的结果卡（即使 Decision 是 BLOCKED），让用户决定是否继续。不要静默进入下一轮。

在 spec 的 Collaboration Summary 中记录讨论轮数和引入的外部角色数量。

## 结果卡

重要讨论必须输出结果卡，包含以下字段：

| 字段 | 必需 | 说明 |
|------|------|------|
| Current Mode | 是 | 当前讨论模式（讨论 / 审阅 / 执行） |
| Decision | 是 | PASS / BLOCKED / REJECT |
| Current State | 是 | 当前状态描述 |
| Why | 是 | 决策原因 |
| Needed Changes | 是 | 需要的改动（如有） |
| Next Step | 是 | 下一步行动 |
| Need Your Input | 否 | 需要用户决策的事项 |
| External Support Request | 否 | 需要外部角色支持的请求 |

## 决策语义

| 决策 | 含义 | 后续 |
|------|------|------|
| PASS | 讨论收敛，可以继续 | 进入下一阶段 |
| BLOCKED | 存在阻塞项，无法继续 | 等待阻塞解除 |
| REJECT | 方案不可接受，需要重新讨论 | 回到讨论，重试计数 +1 |

## 角色简报

讨论结束时，每个参与角色输出简报：

- 当前判断（1-2 句）
- 关注风险
- 下一步建议

角色简报是结果卡的补充，提供各角色视角的细节。

## 思维透镜

讨论中可按问题场景装载思维透镜，帮助看清问题：

- 第一性原理 — 回到基本事实，不依赖类比
- 苏格拉底提问 — 通过追问暴露隐含假设
- 奥卡姆剃刀 — 优先选择最简单的解释

透镜用于辅助思考，不构成强制流程。

## 需求记录

当讨论产出 spec 时，必须将原始需求写入 requirement.md：

- 记录用户的原始表述
- 记录讨论中澄清的细节
- 记录最终收敛的需求范围

## 工作记忆写入

讨论收敛后，将关键决策和上下文写入 workspace.md：

- 讨论中的关键决策和理由
- 被排除的方案及原因
- 需要在执行阶段注意的约束

workspace.md 是跨阶段的共享工作记忆，供后续阶段参考。

## Spec 中的讨论痕迹

由讨论产出的 spec 必须包含：

- 协作摘要（Collaboration Summary）— 参与角色、讨论轮次、关键分歧
- 讨论摘要（Discussion Digest）— 讨论过程的精简记录
