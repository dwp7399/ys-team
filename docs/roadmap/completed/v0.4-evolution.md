# v0.4 演进路线

## Background

v0.3.x 系列（0.3.0 → 0.3.2）交付了：
- 自动编排框架（full-auto/semi-auto）、角色记忆系统、TEAM.md 配置入口
- ai-gateway 治理规则回流（Infer First、Subagent Policy、Spec-Review Gate 等 8 条）
- 发布线治理（`release/<version>` 分支流程 + Release Intent 方法论）
- npm-publish.md + 方法论 06 发布线章节
- Rebuild 补齐 `.ys_team/team.md` 角色结构段

当前版本：**v0.4.0**（已发布 npm）

立项时的三个结构性缺口是：
1. full-auto 编排从未端到端执行过——角色记忆文件全空是直接证据
2. 用户迁移路径断层——`check-update` 只报版本号，不指导"该改什么"
3. 跨仓回流依赖人工分析——没有结构化的回流协议

v0.4 的目标是兑现已有承诺，而不是引入新概念。当前发布线已完成 A1、A2、B1、B2；方向 C 仍保持暂缓。

## Goals

1. 让 full-auto 编排机制真正可用（有实战证据）
2. 让 baseline 演进对用户可感知、可操作
3. 让跨仓经验回流有可重复的结构化路径

## Non-Goals

- 不引入新的用户心智层（无新概念）
- 不做 CI/CD 集成
- 不做 Web UI 或可视化界面

## Directions & Initiatives

### 方向 A：编排实质化（优先级：P0）

**A1：full-auto 端到端验证**
- 目标：在本仓用 full-auto 模式完整跑一个 spec 的 spec-talk → spec-review → spec-work → qa → close 流程
- 验收：`.ys_team/memory/roles/*.md` 至少两个角色有非空经验记录
- 依赖：无
- 输出：验证报告 + 角色记忆初始内容

**A2：Toolbox 第一批内化**
- 目标：为「方法论架构师」和「产品演进负责人」写真实可用的工具内化文件
- 验收：`.ys_team/toolbox/` 有 ≥2 个有实质内容的工具文件；team.md tools 字段引用可验证
- 依赖：A1（工具使用模式在实战中确认后内化）
- 输出：toolbox 工具文件
- 结果：已完成，当前已有 `一致性检查`、`完成前验证` 两个内化工具文件

### 方向 B：用户演进基础设施（优先级：P1）

**B1：Baseline changelog 机制**
- 目标：每次 baseline 重大变更在 `examples/baseline/CHANGELOG.md` 留结构化记录，注明 what changed、why、migration hint
- 验收：v0.4 发布时 CHANGELOG.md 存在且覆盖 v0.3 → v0.4 的变化
- 依赖：无
- 输出：CHANGELOG.md + changelog 维护规范（写入 AGENTS.md）

**B2：check-update 升级 diff**
- 目标：`check-update` 输出 baseline diff 摘要（key files changed）和迁移建议，不只是版本号
- 验收：运行 `npx ys-team check-update` 时，落后版本能看到"以下文件有变化：...建议操作：..."
- 依赖：B1（changelog 是 diff 内容的来源）
- 输出：scripts/ys-team.mjs 升级 + 文档更新

### 方向 C：跨仓学习规范化（P2 · 暂缓）

C1、C2 暂不立项，等需求积累后评估。

## Risks

- A1 验证可能暴露 full-auto 编排逻辑的 gap，需要回头修 spec-talk/spec-work skill
- B2 依赖 npm registry 访问，check-update 本身有网络边界问题（可降级为展示链接）

## Exit Criteria

v0.4.0 发布条件：
- [x] A1 验证完成，有角色记忆记录
- [x] A2 toolbox 首批内化完成
- [x] B1 changelog 存在
- [x] B2 check-update 输出 diff 摘要
- [x] 版本号 bump 到 0.4.0

## Closeout

- v0.4 已按发布线完成收口，当前无新的队列 spec
- `A2` 已落地，但其原始 control 背景已过期；如后续继续做 toolbox 演进，应以新问题重新立项
- `C0` 的载体改动已被 A1 吸收，不再作为独立 initiative 推进
