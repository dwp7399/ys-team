# ys-team v0.4 演进路线

## 当前能力边界（v0.3.1）

### 已具备
- ✅ 完整的 spec-first 工作流（discussion → spec → work → acceptance）
- ✅ 多角色讨论与收敛机制
- ✅ 三种编排模式（manual / semi-auto / full-auto）
- ✅ 角色记忆系统（跨任务经验积累）
- ✅ 现实索引生成（模块关系图 + 业务摘要）
- ✅ npm 分发（全局安装 + 项目级安装）
- ✅ 项目类型检测与本地化（Python/Java/前端/全栈）
- ✅ 文档权威模式与同步规则
- ✅ Spec-Review Gate 和 QA Gate

### 边界
- ❌ 角色记忆仅在单个角色内积累，跨角色经验无法流动
- ❌ 外部角色（安全审查、数据迁移审计等）需要手动判断和引入
- ❌ 深度适配 Claude Code，其他平台（Cursor/Windsurf/Gemini CLI）支持有限
- ❌ 工具箱内化流程存在但未充分验证
- ❌ 演进机制（角色缺口检测 → 评估 → 引入）已设计但未落地

## 演进方向与优先级

### P0：跨角色记忆流动（v0.4.0 目标）

**问题**：
当前角色记忆是孤岛。方法论架构师在某个 spec 中发现的"模板格式问题"，产品演进负责人在下一个 spec 中无法感知。

**目标**：
- 建立团队级知识库（`.ys_team/memory/team/`）
- 角色在讨论结束后，可以将"值得团队共享的经验"写入团队记忆
- 其他角色在后续讨论中可以读取团队记忆

**实现路径**：
1. 设计团队记忆 schema（来源角色、适用场景、经验内容、置信度）
2. 在 spec-talk 收敛后，增加"团队记忆写入"环节
3. 在 spec-talk 启动时，增加"团队记忆读取"环节
4. 在 `.ys_team/memory/policy.md` 中增加团队记忆管理规则

**验收标准**：
- 团队记忆文件存在且格式符合 schema
- 角色在讨论中能引用团队记忆
- 团队记忆有压缩和淘汰机制（避免无限膨胀）

**预计工作量**：1-2 个 spec

---

### P1：外部角色自动判断（v0.4.x 目标）

**问题**：
当前外部角色（安全审查、数据迁移审计、无障碍专家等）需要手动判断和引入。讨论中如果发现角色缺口，需要人工决策是否升级。

**目标**：
- 根据改动类型和影响面，自动判断是否需要外部角色
- 提供角色缺口检测 → 评估 → 引入的完整流程

**实现路径**：
1. 在 spec-talk 中增加"角色缺口检测"逻辑（已有设计，见 skill 文档）
2. 建立外部角色库（`.ys_team/roles/external/`）
3. 设计角色引入触发规则（基于改动类型、文件路径、关键词等）
4. 在 `.ys_team/evolution/requests.md` 中记录角色缺口和评估结果

**验收标准**：
- 角色缺口能被自动检测并记录
- 外部角色库至少包含 3-5 个常见角色（安全审查、数据迁移、无障碍、性能优化、API 设计）
- 角色引入流程可追溯

**预计工作量**：2-3 个 spec

---

### P2：多平台支持（v0.5.0 目标）

**问题**：
当前 ys-team 深度适配 Claude Code（skill 格式、工具名称、编排模式），其他平台支持有限。

**目标**：
- 支持 Cursor、Windsurf、Gemini CLI 等主流 AI coding 工具
- 提供平台适配层，让核心方法论与平台实现解耦

**实现路径**：
1. 调研 Cursor/Windsurf/Gemini CLI 的 agent 能力和限制
2. 设计平台适配层（工具映射、编排模式映射、文件路径约定）
3. 为每个平台提供独立的安装和初始化脚本
4. 在 docs/guide/ 中增加多平台使用指南

**验收标准**：
- 至少支持 2 个非 Claude Code 平台
- 核心方法论文件（docs/methodology/）保持平台无关
- 每个平台有独立的 README 和安装指南

**预计工作量**：3-5 个 spec
