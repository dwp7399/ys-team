# 交付守门人 经验记忆

## evidence 质量要求（2026-04-15）

**错误模式**：evidence 文件只写"验收通过"等成功结论，不记录执行过程中遇到的 gap 和处理方式。

**正确做法**：evidence 必须同时包含：①成功结论及对应验证命令输出，②链路中发现的 gap 及实际处理方式（即使是人工兜底也要写明）。这样后续才能区分"机制真正可用"和"靠人工补救通过"。

**适用场景**：所有 spec 的 QA 和 close 阶段；尤其是首次验证新机制（如 full-auto 模式）时。

## 多 spec 合并发版的处理模式（2026-05-07）

**场景**：一个 release（如 0.6.0）由 N 个 spec 组成时，前序 spec 不应单独走完 close 触发 npm publish，否则会形成 N-1 次中间发版，污染 release 线。

**正确做法**：前序 spec 完成 spec-work + qa 后停在 `docs/specs/active/`，状态写明"等下一个 spec 一起 close"；最后一个 spec 的 close 阶段统一执行 npm publish + 合回 main + tag + push，并在归档时把所有合并发版的 spec 一起移到 archive。

**关键约束**：rules.md 第 67 行"close 前必须 npm publish"在多 spec 合并发版场景下应理解为"release 单元"层面的约束，不是"单个 spec"层面。spec.md 的 AC 必须显式写明"qa PASS 后不进 close"，否则 full-auto 链路会自动触发 close 链路并尝试发版。

**适用场景**：用户在路由阶段明确说"两/多个 spec 合并发版"时；spec 划分以风险隔离为目的而非 release 单元为边界时。
