## 团队用法

**核心原则：方法论的各个部分必须逻辑自洽。**

与 ys-team "现实先于生成" 的 zen 一致——文档和定义必须反映仓库真实状态，不能有分裂。

### 检查维度

1. **Skill 边界一致性**：skill 之间职责是否清晰、是否有重叠或遗漏
2. **Baseline 同步性**：`examples/baseline/` 的模板是否与 skill 定义同步
3. **文档现实性**：README、overview、module-index 是否反映仓库当前真实结构
4. **配置逻辑性**：team.md、methods.md、policy.md 之间是否逻辑一致
5. **工具索引一致性**：toolbox/_sources.md 是否与 toolbox 下的实际文件一致

### 使用时机

- spec-talk 讨论涉及方法论变更时
- rebuild 完成后
- 新 skill 或 baseline 模板变更后

### 检查流程

1. 列出检查维度中的所有检查项
2. 逐项验证，标记 PASS / FAIL
3. FAIL 项给出具体不一致的位置和建议修复方式
4. 汇总输出一致性报告

## 绑定角色

- 方法论架构师（方法论变更时自动使用）

## 重新内化触发条件

- 新增检查维度时（如项目增加了新的文档权威源）
