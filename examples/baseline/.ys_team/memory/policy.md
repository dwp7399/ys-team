# 记忆管理策略

## 结构

```
.ys_team/memory/
  ├── roles/           ← 角色独立记忆（跨任务经验）
  │   └── <role>.md    ← 每个角色一个文件
  └── policy.md        ← 本文件
```

工作记忆（workspace.md）不在此目录，而是跟随 spec 生命周期，位于 `docs/specs/<id>/workspace.md`。

## 角色独立记忆

- 每个角色只读写自己的记忆文件，互不可见
- 记录跨任务的经验积累，不记录当前任务的临时状态
- 长度上限由 TEAM.md 的 `memory.role_memory_limit` 配置（默认 2k）

### 写入时机

每个阶段结束时，角色回顾本次工作，决定是否有值得记住的经验。使用以下判断标准：

1. 是否发现了非显而易见的技术陷阱或模式？
2. 是否有需要在未来同类任务中注意的点？
3. 是否有关于这个项目/模块的特殊知识？

无新认知时不写入。

### 压缩策略

读取记忆时发现超限，结合本次要写入的新内容一起压缩：

- 保留所有仍然相关的经验
- 合并相似条目
- 删除已过时或被后续经验覆盖的条目
- 保留具体的文件路径、错误信息、配置值，不泛化
- 压缩后不超过配置上限

### 记忆格式

每条经验必须使用三段式，禁止纯事实记录：

```markdown
# <role> 经验记忆

## <场景主题>（YYYY-MM-DD）

**错误模式**：<在什么情况下容易犯什么错，越具体越好>
**正确做法**：<应该怎么做，给出可操作的判断规则>
**适用场景**：<什么时候这条规则生效，边界是什么>
```

**禁止**写成纯事件描述（如"本次 spec 发现了 X 问题"）。经验必须能在未来同类场景中直接复用，形式是判断规则而不是事后总结。

## 工作记忆（workspace.md）

- 位于 `docs/specs/<id>/workspace.md`
- 记录当前任务的实时状态：进度、阻塞、决策、各阶段间传递的上下文
- 长度上限由 TEAM.md 的 `memory.workspace_limit` 配置（默认 2k）
- spec 关闭后自动归档（随 spec 目录保留）

## 健康检查

rebuild 时检查：
- 角色记忆文件是否超限
- 是否有角色记忆文件对应的角色已从 TEAM.md 移除（标记为归档，不删除）
- 是否有新角色缺少记忆文件（自动创建空文件）

## 隐私约束

- 不记录个人身份信息（PII）
- 不记录外部系统的密钥、token 或凭证
- 只记录技术经验和项目特有规律
